Visibility Optimization for Static Entities
Overview of the Problem

In a large static world (e.g. voxel terrain with props), Bevy’s default culling iterates every visible entity each frame to perform frustum checks. This per-frame CPU cost can be significant, especially with thousands of static meshes. The goal is to reduce this overhead by precomputing visibility for static geometry and using that data to avoid redundant per-frame checks. Essentially, we want to leverage the static nature of terrain/props to skip iterating over objects that we know can’t possibly be visible from a given camera position or angle.

Precomputing Visibility: Techniques and Feasibility
Potentially Visible Sets (PVS) and Occlusion Maps

One approach is to compute potentially visible sets (PVS) for various regions or camera positions in the world. PVS means determining in advance which chunks or zones could be visible from each other, usually by analyzing occlusion due to static geometry. Classic engines (Quake/Source) precomputed PVS for BSP tree leaf clusters, and modern engines like Unreal and Unity have similar offline occlusion systems. For example, Unreal Engine supports Precomputed Visibility Volumes that bake visibility data into cells covering the level. This stores which actors are visible from each cell, trading runtime culling work for memory and precompute time. Unity’s occlusion culling likewise allows marking objects as static occluders/occludees and baking an occlusion map in the editor. These systems are most effective in mostly static environments with constrained camera movement (e.g. indoor levels or mobile games).

In a voxel world divided into chunks, a PVS approach could mean determining which chunks can see each other. A prominent example is Minecraft’s static cave culling algorithm. Minecraft computes a visibility graph for each chunk by analyzing connections through open faces. Each chunk stores 15 bits encoding which pairs of its 6 faces have line-of-sight through the interior. At runtime, the engine performs a BFS flood-fill from the player’s current chunk, propagating outwards through only those faces marked as visible. Any chunk that cannot be reached via this graph is considered occluded and skipped. This method yielded massive culling benefits – reportedly hiding 50–99% of geometry in complex cave scenes – allowing even mobile hardware to handle Minecraft’s caves. The BFS traversal naturally orders chunks front-to-back and uses the precomputed face-visibilities as a fast occlusion test for neighbors. Only after passing these connectivity tests is a neighbor chunk finally frustum-culled (as the last step). In essence, this produces a conservative PVS per chunk (if the camera is in chunk A, only chunks reachable through the open-face graph are considered visible). This approach can be applied to any grid/portal-like world: you precompute static occluder connectivity, then at runtime drastically restrict the set of chunks/entities that need frustum checks.

Another possible PVS strategy is to partition the world into zones or volumes (rooms, sectors, or just an evenly spaced grid of cells) and precompute which zones are visible from each zone. This is similar to Unreal’s precomputed visibility cells (which are generated around static geometry and store visible actors for each cell). In a custom engine or Bevy plugin, one could divide the map into larger convex regions or grid cells and perform offline raycasts or rasterization from each region to build a lookup of visible static objects. At runtime, you’d find which region the camera is in and enable only the pre-marked visible set. However, the granularity and memory requirements of such a precomputed map can be high if the world or camera space is continuous. Often it’s practical only for fairly static, structured worlds (e.g. indoor levels or discrete areas). For an open-world voxel terrain, a full precomputed visibility table may be infeasible, but the chunk-graph approach (like Minecraft’s) is a specialized form that exploits the grid connectivity of terrain.

Hierarchical and Chunk-Level Culling

A simpler approach than full PVS is to use a hierarchical culling structure (spatial partitioning) for static objects. Instead of checking every object against the frustum each frame, static objects can be organized into a tree (e.g. an octree, BSP, BVH) so that large groups of them can be culled with a single test. For example, one could subdivide the world into an octree of bounding volumes, each node covering a region of space. At runtime, you test the octree nodes against the camera frustum: if a node’s AABB is entirely outside the frustum, you skip all objects in that node, potentially cutting thousands of checks with one test. If a node is partially inside, you recurse into children. This is a well-known broad-phase acceleration technique. A developer discussing a custom culling solution suggested exactly this: bake static objects into an n-ary spatial tree (like an octree) and use it to determine visible volumes and their contents, greatly reducing per-frame work. This approach doesn’t precompute occlusion (just frustum inclusion), but it significantly reduces the number of per-object frustum intersections by culling whole groups at once.

In the context of a chunked voxel game, you already have a natural spatial grouping: the chunk. We can treat each chunk as a unit for culling. For instance, compute a bounding box for each chunk’s static geometry/props (which might just be the chunk’s world-space extents). Then perform frustum culling on the chunk level first. If a chunk’s AABB is outside the view frustum, you skip all entities in that chunk without checking each one. If the chunk is inside or intersects the frustum, then you can consider its contents for rendering. This cell-based culling is a common technique in foliage and large scenes: objects are bucketed into grid cells and culled in groups rather than individually. Using chunk-level culling in Bevy could be as simple as giving each chunk an entity with a GlobalTransform and an Aabb that encompasses all its contents, and only iterating into the chunk’s children if that AABB intersects the frustum. Even without fancy occlusion, frustum-culling at chunk granularity can greatly reduce the cost when many chunks are off-screen (e.g. behind or outside the camera view).

Hierarchical culling can also extend to multiple levels: e.g. chunks grouped into regions, or an octree where leaves are chunks or individual objects. The trade-off is the complexity of maintaining the tree (especially if the world streams in/out or changes). But for static scenes, building an octree once at load time or in a background thread is feasible. This doesn’t eliminate per-frame culling, but it replaces an O(N) iteration over N objects with an O(log N) tree traversal (plus a few checks for objects in visible leaves). It’s effectively caching spatial relationships so that large static groups don’t need to be reconsidered every frame.

Bevy’s Current Capabilities (Visibility APIs)

Out-of-the-box, Bevy’s renderer does frustum culling per entity each frame: it stores an axis-aligned bounding box (Aabb) for each mesh and tests it against the camera frustum, toggling the entity’s ComputedVisibility (or ViewVisibility) accordingly. There isn’t a built-in concept of static vs dynamic visibility optimization beyond this frustum test. As of Bevy 0.16, an experimental occlusion culling feature was added, but it uses a dynamic GPU-based approach rather than precomputed PVS. Specifically, Bevy 0.16+ supports two-phase occlusion culling using hardware depth queries (combined with the new GPU-driven rendering), which can hide objects blocked by others in real time. This is a different philosophy from precomputing static visibility – it happens each frame on the GPU and is more similar to Unity’s runtime occlusion or hardware occlusion queries. In fact, the Bevy 0.16 release notes explicitly contrast their approach with “traditional ‘potentially visible sets’ design”. So currently, Bevy does not provide a built-in PVS or portal system for static geometry; it leans on frustum culling and optional GPU occlusion queries.

That said, Bevy does expose low-level tools that we can leverage for our own visibility scheme. Notably, you can mark entities with the NoFrustumCulling component to opt them out of Bevy’s automatic culling system. This means you can take manual control – for static objects, you might disable Bevy’s per-frame cull and instead use your precomputed logic. Additionally, Bevy’s Visibility component (and its propagation to children) can be used to hide/show groups of entities efficiently. For example, if you have a parent entity for each chunk, toggling that parent’s Visibility::Hidden will instantly remove all its children from rendering without needing to remove them from the world. You could combine this with a custom culling system: e.g., mark all chunk parents with NoFrustumCulling, run your own system to compute which chunk IDs are visible, and set those chunk entities to Visible while hiding the rest. This way, Bevy’s renderer only ever considers the chunks you’ve flagged as visible, skipping the others entirely in draw preparation.

Currently there isn’t an official Bevy plugin for precomputed visibility or octree culling, so you would likely implement this as a custom plugin in your game. However, the community has discussed such needs. For instance, a Bevy issue was raised noting that iterating over all entities for culling “even if done in parallel, is not cheap” and proposing a way to plug in custom visibility logic (for grid-based games, etc.). Until a feature like that lands, the approach is to DIY with the tools above: group static entities, use your own culling criteria, and override Bevy’s default behavior when appropriate.

Trade-offs of Precomputing Visibility

Precomputing visibility for static geometry can yield big performance wins, but it comes with several considerations:

Memory and Precompute Time: Storing PVS data or spatial trees consumes memory. Unreal’s docs note that precomputed visibility trades runtime work for increased memory usage and offline build time. You should restrict precomputed data to needed areas (e.g. don’t generate cells for the entire world if the player only visits part). In a voxel game with procedurally generated terrain, fully offline precomputation might not even be possible unless the world is pregenerated. A dynamic background computation (like Minecraft doing chunk visibility updates on a background thread when blocks change) can amortize the cost. But that adds complexity and potential stutters if not managed carefully.

Camera Movement Flexibility: Precomputed visibility is easiest when the camera/player movement is constrained (e.g. walking through defined corridors or fixed height). If the camera can go anywhere (fly, or free-look 360°, or climb to high vantage points), the PVS has to be conservative enough to account for that. For example, Unreal’s system needs you to specify a “playable area height” to generate cells at the right elevations. In a fully 3D open world, a single PVS per chunk might not capture altitude differences or unusual angles. The Minecraft chunk method assumes the camera is roughly within the chunk volume and considers any angle from that chunk (it was designed for ground-level players in caves). If your game allows very high viewpoints (e.g. spectator mode flying above), previously occluded chunks could become visible. You’d need to either disable the static culling in those cases or precompute additional data for high altitude views (perhaps a separate PVS for sky-high cameras, or just rely on frustum/distance culling when airborne).

Conservativeness vs Accuracy: Precomputed methods generally must be conservative – i.e. never cull something that is actually visible. This often means the PVS might overestimate visibility. For instance, a PVS that is independent of camera orientation will include any object that could be seen from anywhere in the current region, even if your current viewing direction isn’t facing it. That could lead to rendering things behind the camera or just outside the frustum if not combined with a frustum test. (You should still do a final frustum cull on the reduced set of candidates – as Minecraft’s algorithm does in its last step.) Similarly, a chunk-level PVS might assume worst-case (if you’re anywhere in chunk A you might see chunk B), so you might draw chunk B even when the camera is at a spot in A that can’t actually see B. These false positives reduce the culling efficacy but ensure correctness. The benefit is that even an imperfect PVS can vastly cut down the search space. The trade-off is more VRAM/CPU going to some objects that aren’t truly visible at that moment, in exchange for skipping the majority that are definitely not visible.

Occluder Dynamics: By definition, precomputed visibility works for static geometry. If your world has dynamic objects or destructible terrain that can become occluders or open new sightlines, the precomputed data can become invalid. You’d either have to recompute at runtime (as Minecraft does when a chunk’s blocks change) or accept that the PVS might be overly conservative (failing to cull something because an occluder moved or disappeared). For mostly-static worlds this isn’t an issue, but in a voxel game if players can mine big holes or build structures, your occlusion graph must update. This is feasible (the Minecraft approach updates 15-bit masks on chunk edits with ~0.1–0.2ms cost each), but it adds complexity and multi-threading considerations.

Worst-Case Scenarios: Precomputed schemes can degrade in worst-case scenarios. If the world is very open (flat terrain, or a long straight hallway, etc.), the PVS for a region might include nearly everything. You then gain little – you basically did extra work to figure out that everything is visible. For example, the Minecraft dev noted a “worst case” with long ravine structures: the BFS could traverse around a bend and end up marking a huge area visible that, intuitively, a straight-line test might not have (because the BFS can take a curved path). They had to introduce heuristics to limit how far the search goes in such situations. This highlights that PVS methods can sometimes over-cull less than ideal geometry. In contrast, hardware occlusion culling (dynamic) will always cull what’s truly hidden at the moment, but costs runtime effort. Depending on scene complexity, a hybrid might be best: use broad-phase PVS to skip whole regions, then rely on GPU occlusion to catch smaller-scale occlusions within the visible set.

Development Overhead: Building a custom visibility system is non-trivial. You need to debug cases where objects pop in/out incorrectly. Precomputing also means additional tooling or build steps. Unity and Unreal provide nice editors for baking occlusion; in Bevy, you’d be writing your own logic or possibly an external tool to compute visibility sets. Ensure that this added complexity is justified by a measurable performance gain. As Bevy’s docs advise, always profile – if your game isn’t actually bottlenecked by culling cost, a simpler path might suffice.

Designing a Bevy Plugin for Static Visibility

Despite the challenges, it is certainly possible to implement precomputed or hierarchical culling in Bevy for a voxel game. Here we outline a potential design, combining some of the ideas above:

Chunk Entity Setup: Structure your world so that each terrain chunk (and its associated props) is parented to a chunk entity or has a chunk ID. Give the chunk entity a Visibility component and perhaps a custom ChunkVisible marker. All meshes within the chunk remain with default Visibility::Inherited. Also add an Aabb for the chunk as a whole (the bounds of its contents) if you plan to do chunk-frustum culling.

Precompute Static Occlusion Graph: When a chunk is generated or loaded, compute which neighboring chunk faces are connected. You can adopt the Minecraft approach of flood-filling the chunk’s voxels to see which faces “see” each other. Store a bitmask or list of visible directions for that chunk. Essentially, chunk A might know “if looking from the north side, you can exit to its west and south” etc. This requires looking at opaque blocks; if your terrain is solid except air tunnels, this works well. If your world is mostly surface terrain (where line of sight is mostly unobstructed over the top), this graph might be less useful, but you could also incorporate height data to know if a chunk is entirely hidden by a mountain from another.

Runtime Culling System: Write a Bevy system (probably in PostUpdate or a dedicated stage) that runs after chunk loading and before rendering. This system will:

Determine the camera’s current chunk (based on camera position).

Perform a BFS (or DFS) starting from that chunk through the visibility graph. Enqueue neighbors that pass the connectivity test relative to the camera’s entry face and orientation. This will collect a set of chunks that are potentially visible. (Apply frustum culling at chunk level here too for efficiency: e.g., if using chunk AABBs, skip adding a neighbor if its AABB is entirely outside the camera frustum – this combines spatial and occlusion culling).

For all chunk entities in the world, update their visibility: mark those in the visible set as Visible and all others as Hidden. Only these “visible” chunks will have their contents rendered.

Optional Within-Chunk Culling: If chunks are large or contain many objects, you may still do an interior frustum cull. You have two choices: (a) allow Bevy’s normal culling to run on the children within visible chunks, or (b) do a second-pass yourself. Option (a) could be achieved by not using NoFrustumCulling on the mesh entities, only on the chunk parent. In that case, Bevy’s check_visibility will still evaluate each mesh, but only those under a Visible parent are considered (others are effectively skipped because their parent is Hidden). This can work, though you should verify that Bevy’s culling system respects parent visibility (usually it does, as invisible parents mean the child is never rendered). Option (b) is to do your own fine culling: for each visible chunk, iterate its objects and check their AABBs against the frustum, hiding any that are out of view. This duplicates what Bevy does, but on a per-chunk subset (much smaller queries). It might not be necessary if chunks are small enough or if you’re fine rendering the whole chunk when it’s visible (e.g., if each chunk is one mesh like a terrain section, you anyway draw it as a unit).

Visibility Components and Flags: Utilize Bevy’s components wisely. The Visibility component is user-controlled; ComputedVisibility/ViewVisibility is set by Bevy. Since we’re overriding the logic, we rely on setting Visibility on chunk entities. By toggling Visibility::Hidden/Visible on chunk parents, Bevy will propagate that to children (via InheritedVisibility). Those children won’t even be considered for rendering if the parent is hidden. This is efficient. Just ensure that when you hide a chunk, you also perhaps add NoFrustumCulling to its children to avoid Bevy still checking them (or simply trust that hidden status short-circuits the check – which it typically does).

Dynamic Considerations: If your game allows altering terrain, you’d need to update the stored occlusion info for that chunk and possibly its neighbors. This is similar to updating pathfinding grids – it can be done asynchronously. The connectivity flood-fill per chunk (~0.1ms as in Minecraft) can likely be done on a background thread pool when a chunk is modified, then the main thread can swap in the updated visibility data. Until updated, you might temporarily err on the safe side (assume the chunk is fully open) to avoid culling mistakes. Also, if a previously occluded area becomes visible due to a blast creating a hole, you want that to appear – so updating the graph is important for correctness in dynamic worlds.

Fallbacks and Tuning: It’s wise to allow disabling this system at runtime or for certain cameras. For instance, if the camera switches to a free-roam debug mode where it can fly through walls, the occlusion culling might aggressively cull things until the graph catches up (since the camera could “teleport” behind occluders). Having a simple way to turn off the custom culling (or tagging certain cameras to use default culling) can be helpful. Additionally, profile the cost of the BFS and visibility toggling. BFS on a 3D grid of chunks can still visit a lot of chunks if the world is open. The Minecraft developer noted visiting up to ~17,000 chunks in worst cases, though typically much less when occlusion is effective. In Rust/Bevy, 10k entity visibility toggles per frame might be borderline; you may consider throttling updates (e.g., update every few frames) or spatially partitioning the BFS (split among threads).

Integration with Bevy’s Renderer: Because we’re manipulating the Visibility component, this plays nicely with Bevy’s rendering pipeline. As long as our system runs before rendering extract (and after any camera movement), the engine will simply skip invisible chunks. By using NoFrustumCulling on chunk entities (or on all static meshes if completely overriding), we avoid redundant work in check_visibility. Essentially, we replace it with our system for those entities. Meanwhile, dynamic entities (mobs, characters) can still use Bevy’s normal frustum culling or even the new GPU occlusion culling as needed – they won’t be hidden by our chunk system since they might move around occluders (you could exclude non-static entities from being children of chunk entities, or mark them as always visible to default culling). This combined approach lets static geometry be culled via fast precomputed logic, while dynamic objects rely on real-time culling.

Conclusion and Example Use-Case

For a voxel game with chunked terrain and mostly static props, implementing precomputed visibility can significantly reduce CPU overhead. You can cache static visibility information in forms like connectivity bitmasks or hierarchical bounding volumes, and use that to avoid per-frame checks on every object. The example of chunk-to-chunk visibility sets (as used in Minecraft) shows that even in a dynamic camera scenario, a conservative precomputed PVS can cut down on over 90% of rendering work in occluded scenes. The trade-off is the upfront computation and complexity, but in scenes with heavy occlusion (dense forests, caves, city streets) it pays off by skipping entire areas that the camera cannot see behind obstacles. In more open scenes, simpler techniques like frustum-culling at the chunk or cell level might suffice, grouping many small objects into one test.

In summary, Bevy does not provide a turnkey static occlusion system, but it gives you the building blocks (visibility components, culling toggles, and an ECS to schedule your systems) to integrate one. You might start with hierarchical frustum culling (e.g. an octree or chunk grouping) to gain immediate wins in culling cost with relatively low complexity. If further optimization is needed, explore a PVS-style plugin for your chunks: precompute which chunks can see which others and use a BFS at runtime to determine the visible set per camera. This could be provided as a VisibilityOptimizationPlugin in your docs/visibility_optimization.md, complete with code examples on how to mark entities and how to hook into Bevy’s update loop. By doing so, your voxel engine can maintain high performance even as the world complexity grows, ensuring that the rendering pipeline only deals with the necessary chunks and objects each frame, and all the fully hidden scenery stays out of the picture (literally!).
